Session 01
GKE — Google Kubernetes Engine
92% Interview Weight

Standard vs Autopilot

Standard
Autopilot
  • You manage node pools, OS, SSH
  • Full DaemonSet support
  • Fine-grained cost control
  • Use: custom nodes, DaemonSets, large-scale cost tuning
  • Google manages all nodes
  • No user DaemonSets allowed
  • Pay per pod resource request
  • Use: new workloads, compliance, faster time-to-prod
Always pick Autopilot for new workloads UNLESS you need DaemonSets, custom OS, or SSH access to nodes.

Workload Identity — Know This Cold

Problem: JSON key files in pods. They leak, never rotate, violate least privilege.

Solution: Bind a Kubernetes Service Account (KSA) to a GCP Service Account (GSA). The GKE metadata server exchanges the KSA token for a short-lived GCP token. No key file anywhere.

  • 01Enable workload pool on cluster
  • 02Bind KSA to GSA with roles/iam.workloadIdentityUser
  • 03Annotate KSA with GSA email
Workload Identity is GKE-specific because K8s needs per-POD identity. A node runs 100 pods — they can't all share the node's SA. Cloud Run, Compute Engine just attach an SA directly.

HPA vs VPA vs Cluster Autoscaler

HPA
VPA
Cluster Autoscaler
  • Adds/removes pod REPLICAS
  • Based on CPU, memory, custom metrics
  • For stateless services
  • Never use with VPA Auto simultaneously
  • Changes CPU/memory REQUESTS
  • Restarts pods to apply changes
  • For batch jobs, right-sizing
  • Use Off mode alongside HPA
  • Adds/removes NODES
  • Triggered by unschedulable pods
  • Node level — not pod level
  • All three can be active together
HPA + VPA Auto = infinite conflict. (HPA-FLAPPING) VPA changes requests → HPA recalculates utilization → scales → VPA changes again. Use VPA in Recommendation (Off) mode with HPA.

Services — The Four Types

  • CIClusterIP — internal only, stable virtual IP. How pods talk to each other.
  • NPNodePort — exposes on 30000–32767 on every node. Rarely used directly in prod.
  • LBLoadBalancer — creates ONE GCP L4 LB per service. 20 services = 20 LBs = expensive.
  • HLHeadless (clusterIP: None) — DNS returns pod IPs directly. Use for StatefulSets (Kafka, Cassandra).
LoadBalancer creates one GCP LB per service. Use Ingress or Gateway API as a single entry point routing to ClusterIP services internally. 20 LoadBalancer services = 20 LBs = unnecessary cost.

Pod Disruption Budgets

PDB answers: during a disruption, how many pods can be down simultaneously?

minAvailable: 2 = always keep 2 pods running. maxUnavailable: 1 = never have more than 1 pod down.

PDB + rolling update math: 4 replicas, maxUnavailable: 2, PDB minAvailable: 3. Update tries to kill 2 → only 2 left → violates PDB → rollout HANGS. Always check: replicas minus maxUnavailable must be ≥ minAvailable.

Ingress vs Gateway API

Ingress
Gateway API
  • L7 HTTP routing, vendor annotations
  • No native traffic weighting
  • One LB for all services
  • Legacy — still widely used
  • GatewayClass / Gateway / HTTPRoute split
  • Native traffic weighting for canary
  • Supports TCP/UDP, not just HTTP
  • Infra team owns Gateway, app team owns HTTPRoute
Use Gateway API for canary — HTTPRoute weight splitting is native. Ingress requires annotation hacks that differ per vendor.

Key Debugging Commands

  • kubectl get endpoints svc-nameFIRST command for any connectivity issue. Empty = no pods match selector.
  • kubectl get networkpolicy -n nsSilent packet drops. Firewall is invisible by design.
  • kubectl describe hpaCheck ScalingActive=False and metrics errors.
  • kubectl top podsRequires Metrics Server. If it fails, Metrics Server is down.
Session 02
VPC & Networking
88% Interview Weight

The 3 Facts Unique to GCP Networking

  • 01VPC is GLOBAL, subnets are REGIONAL. One VPC spans all regions. VMs in different regions on the same VPC communicate via internal IPs with no extra config.
  • 02Firewall rules are VPC-WIDE, applied via network TAGS on VMs — not attached to subnets. Write one rule, applied to every tagged VM everywhere in the VPC.
  • 03Every subnet talks to every other subnet BY DEFAULT. Absence of a firewall rule = allow. Use firewall rules to restrict, not to enable.

Firewall Rules — Priority Wins

Direction: INGRESS or EGRESS. Action: ALLOW or DENY. Priority: 0–65534. Lower number = higher priority.

Priority 800 DENY beats Priority 1200 ALLOW. Lower number ALWAYS wins regardless of specificity. Default rules sit at 65534 — any rule you add overrides them.

Cloud NAT vs Private Google Access

Private Google Access
Cloud NAT
  • For: GCP APIs (BigQuery, GCS, Secret Manager)
  • Enabled: per SUBNET — one flag
  • Traffic stays on Google backbone
  • No Cloud Router needed
  • Use: any VM/pod needing GCP APIs without external IP
  • For: general internet outbound (apt-get, Stripe API)
  • Requires: Cloud Router first, then NAT gateway
  • Outbound ONLY — no inbound connections possible
  • One router+NAT covers all subnets in a region
  • Use: VMs with no external IP needing internet egress
PGA = GCP API back door per subnet. Cloud NAT = outbound internet post office per region. (OR MULTI-REGION) Both often used together. Neither enables inbound connections from internet.

VPC Peering vs Shared VPC

VPC Peering
Shared VPC
  • Connects two VPCs directly
  • Non-transitive: A–B, B–C does NOT mean A–C
  • Each VPC independently managed
  • For: different orgs, teams needing full autonomy
  • CIDR ranges must NOT overlap
  • Host project owns VPC, subnets, firewall rules
  • Service projects deploy VMs into host VPC
  • App team cannot touch network config
  • For: same org, central network governance
  • Solves governance — NOT environment isolation
No peering = environment isolation (prod can't reach dev). Shared VPC = network governance (app team can't misconfigure prod). Two completely different problems. You need both in a fintech setup.

OS Login

Links SSH access to Google identity instead of raw key files. Offboarding = suspend Google account → SSH revoked on ALL VMs instantly.

roles/compute.osLogin → SSH without sudo roles/compute.osAdminLogin → SSH with sudo enable-oslogin-2fa=TRUE → MFA constraints/compute.requireOsLogin → Enforce everywhere
OS Login → short-lived certificates tied to Google identity. No key to rotate, no key to leak, no manual offboarding. The CISO's favourite feature.
Session 03
IAM
85% Interview Weight
IAM = WHO (identity) can do WHAT (role) on WHICH (resource). Everything in IAM is a binding of these three things.

Primitive vs Predefined vs Custom Roles

Primitive — AVOID
Predefined — USE THESE
Custom — Advanced
  • roles/owner, roles/editor, roles/viewer
  • Project-wide, overprivileged
  • Never in production
  • roles/storage.objectViewer
  • roles/bigquery.dataViewer
  • roles/container.developer
  • Service-specific, granular
  • Curate exact permissions
  • Maintenance overhead
  • GCP won't auto-add new permissions
  • Use sparingly when predefined doesn't fit
dataViewer alone cannot run BigQuery queries. Need jobUser too — at PROJECT level (not dataset level, because jobs are project-level resources). Always grant both.

Service Accounts

A service account is both an IDENTITY (grant it roles to access resources) and a RESOURCE (grant roles ON it to allow impersonation).

  • UAUser-managed SA: you create. Use for your workloads.
  • DCDefault Compute SA: auto-created with roles/editor — DANGEROUS. Always restrict or replace.
  • GMGoogle-managed SA: internal GCP use. Never touch these.
Default Compute Engine SA has roles/editor on the project. Every pod on that GKE node inherits this. One compromised pod = full project access. Always create a minimal custom SA for node pools.

Workload Identity Federation — External Systems

GitHub Actions / AWS / Azure needing GCP access. Old way: SA key file → leaks, never rotates. New way: WIF exchanges OIDC tokens for short-lived GCP tokens. No SA keys anywhere.

Org Policy vs IAM

IAM
Org Policy
  • Controls WHO can perform actions
  • Identity-based
  • Can be granted/revoked
  • Example: Alice has no permission to create VM with external IP
  • Controls WHAT configurations can exist
  • Resource-based, API-level enforcement
  • Trumps IAM — even roles/owner cannot bypass
  • Example: NOBODY can create VM with external IP
Org Policy > IAM. Always. For hard requirements (no external IPs, no SA keys, data residency) use Org Policy — enforced at the API level, unforgeable.

Audit Logs

  • AAAdmin Activity — always on, free. IAM changes, resource creation/deletion.
  • DAData Access — off by default, costs money. Who read secrets, queried BQ, accessed GCS. Enable for PCI/GDPR/SOC2.
  • SESystem Event — always on, free. GCP internal operations (live migration, auto-scaling).

Key Org Policy Constraints

  • constraints/compute.vmExternalIpAccess→ DENY — no external IPs on VMs
  • constraints/iam.disableServiceAccountKeyCreation→ ENFORCE — force Workload Identity
  • constraints/compute.requireOsLogin→ ENFORCE — no raw SSH key files
  • constraints/gcp.resourceLocations→ allowedValues — GDPR data residency
Session 04
BigQuery
75% Interview Weight
BigQuery is NOT a database. It is a serverless columnar analytical engine (data warehouse). It scans columns, not rows. It charges per byte scanned, not per query time. This single fact drives every design decision.
SELECT * FROM huge_table = scans EVERY column = massive bill. Always SELECT only the columns you need. The single most common BigQuery cost mistake in every organisation.

Partitioning vs Clustering

Partitioning
Clustering
  • Physically splits data into segments
  • Query skips entire partitions (pruning)
  • 1 column max: time / range / ingestion
  • Exact cost estimate before running
  • Set requirePartitionFilter=true on large tables
  • Sorts blocks within each partition
  • Query skips non-matching blocks
  • Up to 4 columns
  • Approximate cost estimate before running
  • Always combine with partitioning for double pruning
Partition by date, cluster by user_id. Filter on both = skip entire date partitions first, then skip blocks within the remaining partition. Scan 2GB instead of 10TB.

Views vs Materialized Views

Regular View
Materialized View
  • Saved SQL query — no stored data
  • Re-executes full query every time
  • Always fresh, always slow on large tables
  • Use for: simple abstractions, small data
  • Pre-computed, stored result
  • Auto-refreshed on schedule
  • Fast and cheap to query
  • Can serve stale data between refreshes
  • Use for: heavy aggregations, dashboards

Streaming vs Batch

Streaming Inserts
Batch Loads (from GCS)
  • Available in seconds
  • $0.01 per 200MB inserted — costs money
  • For: real-time dashboards, event pipelines
  • No updates/deletes for 30 min after insert
  • FREE — no charge
  • CSV, Parquet, Avro, ORC
  • For: ETL pipelines, daily data loads
  • Prefer batch unless real-time is genuinely required

Cost Control

  • bq query --dry_runSee bytes scanned before spending a cent
  • --maximum_bytes_billed=1000000000Safety net — query fails if it would exceed limit
  • requirePartitionFilter=trueAnalysts cannot accidentally full-scan large tables
  • INFORMATION_SCHEMA.JOBS_BY_PROJECTFind expensive queries daily

Slots & Reservations

One slot = one unit of compute doing query work. On-demand: up to 2,000 shared slots per project. Reservations: buy dedicated slots, divide between teams via assignments.

Slow queries during business hours, fast at night = slot contention. You're competing in the shared pool. Buy a reservation and assign it to your project for guaranteed performance.
Session 05
Pub/Sub
70% Interview Weight

Pub/Sub = asynchronous messaging. Producer publishes once. Each subscription gets its own independent copy. One slow consumer never affects others.

  • TTopic: the channel. Publishers send here.
  • SSubscription: how consumers receive. Each gets ALL messages independently.
  • FOFan-out: one topic + N subscriptions = every message goes to all N consumers. Add a 6th subscriber = zero code change in publisher.

Push vs Pull

Pull
Push
  • Consumer asks for messages when ready
  • Natural backpressure — pull when you can process
  • For: GKE workers, Dataflow, high-throughput batch
  • Consumer controls the rate
  • Pub/Sub delivers to HTTPS endpoint
  • For: Cloud Run, Cloud Functions, event-driven serverless
  • Pub/Sub controls the rate
  • 200–204 = ACK · 4xx/5xx = NACK → retry
Push has no natural backpressure. If your endpoint is slow, Pub/Sub keeps pushing. Pull naturally handles backpressure — consumer controls what it can handle.

ACK, NACK, ACK Deadline

After pulling a message, the ACK deadline clock starts (default 10s). Must ACK or NACK before expiry. Deadline expires → Pub/Sub redelivers → duplicate processing.

  • ACKProcessed successfully, delete it
  • NACKFailed temporarily, redeliver immediately
  • ∅Deadline expires → automatic redeliver to any available subscriber
NACK on a permanent error = infinite retry loop. ACK permanently broken messages, then handle the error separately (log, DLQ, alert). Never NACK something that will always fail.

Dead Letter Topics & Ordering

  • DLTAfter N failed delivery attempts, message moves automatically. Always set max-delivery-attempts (e.g. 5). Monitor: a spike = something broke in your consumer.
  • ORDOrdering keys: same key = same order delivered. Enable --enable-message-ordering on subscription.
  • HOLHead-of-line blocking: failed message with ordering key blocks ALL subsequent messages with same key.
  • EOExactly-once: enable per subscription, pull only, higher cost. Use for financial transactions.
Session 06
Cloud Storage & Databases
68% Interview Weight

The Decision Flowchart

Step 1Need SQL (joins, ACID)? YES → Cloud SQL or Spanner. NO → Firestore, Bigtable, or Memorystore.
↓
Step 2If SQL — Need global scale or horizontal write scaling? YES → Spanner. NO → Cloud SQL (10× cheaper).
↓
Step 3If NoSQL — Documents / flexible schema → Firestore. Time-series / high throughput → Bigtable. Just caching → Memorystore.

The Database Matrix

Cloud SQL
  • Managed MySQL/PostgreSQL
  • Single region, vertical scale
  • HA = standby replica (not readable)
  • Read replica = readable, for scale
  • For: web apps, CMS, ERP, <10TB
Cloud Spanner
  • Global SQL + ACID + horizontal write scale
  • Virtually unlimited scale
  • ~$2,000/month minimum — expensive
  • For: fintech, global inventory, multi-region
  • Never for: small or cost-sensitive workloads
Firestore
  • Serverless NoSQL document store
  • Real-time client sync — UNIQUE to Firestore
  • Offline support built-in
  • For: mobile/web apps, user profiles, game state
  • Limit: 1 write/sec per document
Bigtable
  • Petabyte-scale wide-column NoSQL
  • Time-series, IoT sensor data, ML features
  • Row key design is EVERYTHING
  • No SQL, no joins, no ad-hoc queries
  • For: sensor readings, financial tick data
Memorystore
  • Managed Redis / Memcached
  • Microsecond latency reads
  • NEVER primary storage — cache only
  • Always have a durable backing store
  • For: sessions, API cache, rate limiting
GCS Storage Classes
  • Standard: hot, frequent access
  • Nearline: monthly access
  • Coldline: quarterly access
  • Archive: yearly, highest retrieval cost
  • Lifecycle policies automate transitions
Bigtable row key: sequential timestamps as the ONLY key = all current writes hit one tablet = hotspot. Use device_id#reverse_timestamp to distribute writes AND support time-range queries per device.

GCS Key Concepts

  • UBAUniform bucket-level access: always use. IAM is the single control. Fine-grained ACL = legacy, avoid.
  • OVObject versioning: every overwrite creates a new version. Add lifecycle policy to expire old versions.
  • RLRetention locks: prevent deletion before duration — even bucket owners can't delete. WORM compliance.
  • LCLifecycle policies: set on day 1. Standard → Nearline (30d) → Coldline (90d) → Delete (365d).
Session 07
Compute Engine & Cloud Run
62% Interview Weight

The Compute Decision

Compute Engine
GKE
Cloud Run
  • Full OS control needed
  • GPU / TPU / custom hardware
  • Legacy apps, can't containerize
  • MIG for VM autoscaling
  • Spot VMs: 80% cheaper, fault-tolerant
  • Containerized + K8s features
  • DaemonSets, StatefulSets, PVCs
  • Complex microservices platform
  • Long-running services
  • When Cloud Run isn't enough
  • Stateless containers, serverless
  • Event-driven or HTTP services
  • Zero infra management
  • Scale to zero = zero cost at idle
  • Simplest operational model

Machine Families

  • E2Economy, cheapest. Dev/test, low-traffic apps.
  • N2Balanced. Most production web workloads, app servers. Sweet spot.
  • C2Compute-optimised. Gaming servers, HPC, CPU-bound batch.
  • M3Memory-optimised, up to 12TB RAM. SAP HANA, large in-memory analytics.
  • A2GPU/TPU. ML training, CUDA workloads.

Cloud Run — Key Concepts

Cold Starts
Concurrency
  • min-instances=0: scales to zero, zero idle cost
  • First request after idle = 1–10s latency spike
  • Fix: min-instances=1 (keep one always warm)
  • For prod APIs: always set min-instances=1
  • Reduce: smaller images, lazy dependency loading
  • One instance handles N simultaneous requests
  • I/O-bound (DB queries, API calls): concurrency=80+
  • CPU-bound (image processing): concurrency=1
  • High concurrency = fewer instances = lower cost
  • Default: 80
concurrency=1000 on a CPU-bound service = 1000 requests sharing one CPU = overloaded instance. CPU-bound: concurrency=1, let autoscaler add instances instead.

Cloud Run Jobs vs Services

Cloud Run Service
Cloud Run Job
  • Long-running HTTP server
  • Scales with incoming requests
  • Never 'finishes' — always listening
  • For: APIs, webhooks, microservices
  • Runs to completion, then stops
  • Batch processing, migration, reports
  • Billed only for actual execution time
  • Replace cron VMs: 10min job on 24h VM = waste

Spot VMs

Up to 80% cheaper than standard VMs. GCP can preempt with 30-second warning. Use for fault-tolerant workloads: ML training with checkpoints, batch data processing, CI/CD workers.

Never use Spot VMs for: databases, user-facing services, or anything that cannot survive sudden termination.
Session 08
Cloud Build & CI/CD
55% Interview Weight

The Pipeline Flow

01Code push → Cloud Build trigger
↓
02cloudbuild.yaml steps: build, test, scan, push
↓
03Artifact Registry → Binary Authorization attestation
↓
04Cloud Deploy → GKE / Cloud Run / Terraform

cloudbuild.yaml — Key Patterns

  • 01Each step runs in a Docker container — any public or private image is a valid build step
  • 02Built-in substitution vars: $PROJECT_ID · $COMMIT_SHA · $BRANCH_NAME · $TAG_NAME
  • 03Parallel steps: waitFor: ['-'] runs steps simultaneously — useful for parallel test suites
  • 04Machine types: options.machineType: E2_HIGHCPU_8 for faster builds
Never tag images as 'latest' in CI/CD. Use $COMMIT_SHA. 'latest' is mutable — you can't tell what's running in prod. $COMMIT_SHA is immutable — full traceability, safe rollback, audit compliance.

Three-Pipeline Pattern

  • PRPR opened → CI pipeline: lint, unit tests, build image, scan. NO deploy. Fast feedback for developers.
  • MNMerge to main → Staging pipeline: build, Binary Authorization attest, deploy staging, integration tests.
  • TAGGit tag v1.2.3 → Release pipeline: pull already-built image, canary deploy to prod via Cloud Deploy, monitor.

Deployment Strategies

Rolling
  • Replace pods gradually one by one
  • Both versions live simultaneously
  • Zero downtime, built into K8s
  • Slow rollback (pod by pod)
  • For: most stateless service updates
Blue-Green
  • Two identical environments
  • Instant all-or-nothing traffic switch
  • Instant rollback (flip selector back)
  • 2× infra cost during switchover
  • For: breaking schema changes, type changes
Canary
  • Gradual traffic shift 10→50→100%
  • Real user traffic validation
  • Instant rollback (zero canary weight)
  • Cloud Run: built-in revision weights
  • Default choice for most deployments
Default to Canary. Switch to Blue-Green ONLY when v1 and v2 absolutely cannot coexist — breaking schema changes (column rename, type change), breaking API contracts.

Expand/Contract — Make Schema Changes Canary-Safe

  • ①EXPAND: add new column, keep old. Deploy v2 writing to BOTH. v1 still works. Canary deploy safely.
  • ②MIGRATE: background job copies data from old column to new.
  • ③CONTRACT: retire v1. DROP old column in a separate deployment.
Expand/Contract = 3 deployments instead of 1, but zero downtime and zero risk. Always prefer over blue-green if you have time to plan it.

Binary Authorization

Cloud Build creates a cryptographic attestation after tests and vulnerability scans pass. GKE admission controller checks for valid attestation at deploy time. No attestation = deploy rejected.

Binary Authorization = supply chain security. Even a developer with full kubectl access cannot deploy an unattested image. The attestation is tied to the image digest — unforgeable without the signing key.

Artifact Registry & Terraform

  • URLImage URL: LOCATION-docker.pkg.dev/PROJECT_ID/REPO/IMAGE:COMMIT_SHA
  • CLNSet cleanup policies on day 1 — without them, builds accumulate and storage costs explode
  • SCANEnable vulnerability scanning: automatic on push when configured
  • TFTerraform remote state: always GCS bucket, never local. Enables team collaboration and state locking.
  • PRPlan on PR, apply on merge — humans review infra diffs before changes happen

Decision Rules — Know These Instinctively

  • →Workload needs GCP API access → Workload Identity (GKE) or SA attachment (Cloud Run/CE)
  • →External system needs GCP access → Workload Identity Federation. No SA keys ever.
  • →Need SQL + global scale → Spanner. Need SQL + single region → Cloud SQL (10× cheaper).
  • →Need real-time client sync → Firestore. Only GCP service with this built-in.
  • →Need petabyte time-series → Bigtable. Design row key carefully — it is everything.
  • →New stateless HTTP service → Cloud Run. Complex stateful platform → GKE.
  • →Batch job replacing a cron VM → Cloud Run Jobs. Pay for execution, not idle time.
  • →DB schema change → Expand/Contract first. If impossible → Blue-Green.
  • →Everything else deployment → Canary. Always. It is the safest choice.
